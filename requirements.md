# Amazon Seller Central Summary PDF 解析需求文档

## 1. 背景

该脚本的目标是批量读取 Amazon Seller Central 的月度 Summary PDF，对账单中的收入、费用、税费、转账等字段进行解析、映射和汇总，最终输出为固定结构的 Excel，供财务入账或后续数据处理使用。

当前样本中存在两类 PDF：

- `*MonthlySummary*.pdf`
- `*MonthlyUnifiedSummary*.pdf`

两类文件都属于单页、双列表格类账单，但字段覆盖范围和版式细节存在差异。系统需要同时支持这两类文件，且在批量处理时能够完整跑通。

## 2. 核心目标

系统必须满足两个最高优先级目标：

1. 批量处理可以完整跑通，不因单个异常字段或单个文件中断整批任务。
2. 输出数据必须正确，尤其是字段归类、金额、月份、店铺信息、国家站点、汇率换算结果必须可信。

## 3. 输入

输入为一个目录，目录中包含 1 个或多个 Amazon Seller Central Summary PDF 文件。

输入文件的已知特征：

- 文件格式为 PDF。
- 当前样本均为单页 PDF。
- 页面主体为左右双列布局。
- 账单中至少包含以下一级分类中的若干项：
  - Income
  - Expenses
  - Tax
  - Transfers
- 头部区域包含以下关键信息：
  - Display name
  - Legal name
  - Account activity / 账期
  - Currency / 币种

## 4. 输出

输出为一个 Excel 文件，Sheet 名为 `AMZ`，列结构固定为：

- 返回目录
- 平台
- 店铺号
- 店铺公司
- 归属月份
- 来源数据报告
- 归属
- 字段
- 解释
- 利润表项目
- 原币金额
- 汇率
- 换算本币（人民币）

## 5. 业务需求

### 5.1 文件处理范围

- 系统应扫描输入目录下所有 `.pdf` 文件。
- 系统应逐个解析文件，并将成功结果汇总输出到同一个 Excel 中。
- 系统必须支持 `MonthlySummary` 和 `MonthlyUnifiedSummary` 两类样本。

### 5.2 头部信息提取

每个 PDF 必须尽可能准确提取以下信息：

- 店铺展示名 `display_name`
- 店铺法务主体 `legal_name`
- 账期文本 `time`
- 币种 `currency`

要求：

- 优先使用 PDF 中真实提取的头部信息。
- 不允许用固定默认值覆盖真实提取出的店铺名或公司名。
- 如果个别头部字段缺失，应记录失败原因或告警，而不是悄悄填错。

### 5.3 国家/站点识别

系统需识别账单所属站点，当前范围至少包括：

- 美国站
- 加拿大站

识别依据优先级建议如下：

1. PDF 中币种信息
2. 文件名中的明确国家信息
3. 配置中的显式映射

要求：

- 站点识别规则必须可解释、可配置。
- 不能使用高误判风险的模糊子串规则作为唯一依据。
- 当站点无法识别时，应将该文件标记为失败，并输出明确原因。

### 5.4 财务明细提取

系统需从双列表格中提取一级分类和二级明细项目。

要求：

- 能识别四大类：`Income / Expenses / Tax / Transfers`。
- 能识别类目下的二级字段名称。
- 能识别该字段的借方/贷方金额，或统一为业务定义下的原币金额。
- 能正确忽略以下非业务数据：
  - 表头
  - 页脚
  - Totals / Subtotals
  - 页面装饰线、分隔线、占位符文本
  - 说明性文本

特别要求：

- `=========================================---- - - -` 这类分隔线必须视为噪音，不得作为业务字段进入结果。
- 对 Unified Summary 中新增字段不能直接判错退出，应先区分“噪音字段”和“真实新字段”。

### 5.5 字段映射

每条有效业务字段都需要映射到目标财务口径：

- 解释
- 利润表项目

要求：

- 映射规则应支持按 `国家 + 分类 + 字段名` 精确匹配。
- 允许配置通用兜底映射。
- 对未映射字段的处理要分级：
  - 噪音字段：直接过滤
  - 可忽略字段：记录 warning，不中断整批
  - 关键业务字段：标记该文件失败，并给出明确原因

### 5.6 金额口径

每条明细应输出一个 `原币金额`。

要求：

- 金额方向定义必须明确，不能仅凭“借贷两列直接相加”这种不带业务语义的方式长期使用。
- 需要给出统一规则，例如：
  - 借方记负，贷方记正
  - 或按 Amazon 报表口径转换为净额
- 同一字段在不同账单类型中应保持一致口径。

如果暂时无法完全依赖账单结构自动判断，则必须在需求实现中明确写出金额规则，并通过样本校验。

### 5.7 月份识别

系统需输出 `归属月份`，格式应为该账期所在月份的月初日期。

要求：

- 优先从 `Account activity from ... through ...` 中提取月份。
- 文件名可作为兜底来源。
- 输出结果应统一为当月 1 号的日期对象或等价日期格式。

### 5.8 汇率与人民币换算

每条明细应包含：

- 原币金额
- 汇率
- 换算本币（人民币）

要求：

- 汇率来源必须明确。
- 如果汇率为固定配置值，应在配置中维护，并允许后续按月份或站点更新。
- 人民币金额计算规则为：
  - `换算本币（人民币） = 原币金额 * 汇率`
- 保留精度规则需要明确并统一。

## 6. 批量处理与容错需求

### 6.1 批处理行为

- 系统不得因为单个文件失败而退出整个任务。
- 所有文件都应尝试处理到底。
- 最终输出中应区分：
  - 成功文件数
  - 失败文件数
  - 总输出行数

### 6.2 异常输出

系统应输出失败明细，至少包含：

- 文件名
- 失败阶段
- 失败原因

建议增加：

- 未映射字段清单
- 被过滤噪音字段清单
- 统计摘要

### 6.3 结果产出策略

即使存在部分失败文件，也应满足以下策略之一：

1. 输出成功文件的数据 Excel，同时输出失败清单。
2. 输出 Excel 的同时在附加 Sheet 中记录错误。

除非全部文件都失败，否则不应完全没有结果文件。

## 7. 正确性标准

“数据是对的”至少应满足以下标准：

- 店铺号、店铺公司应与 PDF 头部信息或业务配置一致，不能误填固定默认值。
- 国家站点识别正确。
- 归属月份正确。
- 每条字段的归属分类正确。
- 业务字段不应混入分隔线、Subtotal、说明文本等噪音。
- 金额方向与净额逻辑一致。
- Excel 中每一行都能追溯回原始 PDF 的一个真实字段。

## 8. 验收标准

以当前目录内 15 个样本 PDF 为验收基线，至少应达到：

1. 所有样本文件都能被处理完成，任务不因中途单文件异常而退出。
2. `MonthlySummary` 与 `MonthlyUnifiedSummary` 两类文件都能正确识别。
3. 不再出现分隔线被当成字段的情况。
4. 输出 Excel 中 `店铺号`、`店铺公司` 不得被错误固定写成默认值。
5. 未映射字段必须可解释，并区分是新业务字段还是噪音。
6. 最终结果可供财务直接核对，至少支持按文件追溯解析来源。

## 9. 建议的数据校验机制

为保证“跑通”和“数据对”同时成立，建议实现以下校验：

- 文件级校验
  - 是否成功提取头部信息
  - 是否识别到主要分类
  - 是否存在异常噪音字段

- 字段级校验
  - 字段名是否为空
  - 字段是否命中噪音规则
  - 字段是否命中映射规则

- 结果级校验
  - 是否存在重复异常行
  - 是否存在明显错误的金额值
  - 是否存在空店铺、空月份、空汇率

## 10. 非功能要求

- 代码应具备可维护性，字段映射和站点配置应尽量从逻辑中解耦。
- 对新增站点、新字段、新语言标签应具备可扩展性。
- 日志输出应能帮助快速定位是哪一个 PDF、哪一个字段出了问题。

## 11. 当前版本已确认的问题

基于真实样本运行，当前脚本已确认存在以下偏差：

- 会把 Unified Summary 中的分隔线识别为业务字段。
- 对未映射字段采用“直接失败”策略，导致整批任务中断。
- 单个文件失败时直接退出，不符合批处理要求。
- 输出中的 `店铺号`、`店铺公司` 被固定默认值覆盖，导致结果不准确。
- 国家识别规则存在误判风险。
- 金额计算规则缺少明确业务定义。

## 12. 后续实现建议

建议按以下顺序落地：

1. 先修正批处理策略，保证所有文件都能跑完并产出结果。
2. 再修正噪音字段识别，解决 Unified Summary 失败问题。
3. 再校正店铺信息、国家识别和金额口径，解决“数据必须对”的问题。
4. 最后补充失败清单、校验日志和可配置映射，便于长期维护。
